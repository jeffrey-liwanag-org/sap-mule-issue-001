MuleSoft ↔ SAP Connectivity

Stale Connection Pool During SAP Zero-Downtime Maintenance

Issue Analysis, Gap Assessment & Proposed Resolution

Current Situation

Our MuleSoft applications integrate with SAP using the MuleSoft SAP Connector, which internally uses the SAP JCo (Java Connector) library to make RFC calls.

How SAP-Side Maintenance Works Today

The SAP Basis team performs system maintenance using a Zero-Downtime / Downtime-Optimized approach. Instead of bringing the entire SAP system offline, individual application server instances are recycled in a rolling fashion behind a message server / logon group.

From the SAP perspective: The system as a whole remains available throughout the maintenance window — there is no formal outage.

What We Observe on the MuleSoft Side

  • RFC calls from Mule begin failing intermittently or persistently after the maintenance window.
  • Failures do not self-heal — they continue indefinitely.
  • The only known recovery today is restarting the Mule application, which forces a fresh connection pool to be built.

The Predicament

Why This Is a Problem

Operational Impact

Manual application restarts are required after every SAP maintenance event — adding toil and delay to recovery.

Reliability Impact

Integrations appear unreliable to downstream consumers despite SAP itself being technically "available."

Detection Gap

The failures often surface only when a consumer reports them — there is no proactive recovery path.

Scaling Concern

As more integrations are added, the blast radius of each maintenance window grows.

The Underlying Behavior

The JCo library maintains a client-side connection pool of long-lived TCP connections to SAP. When SAP recycles an app server during ZDM, those pooled connections become stale — but JCo does not always detect this proactively.

Result: every borrow from the pool returns a broken connection. Restarting Mule rebuilds the pool — which is why the restart "fixes" it.

Technical Root Cause

How JCo's Connection Pool Behaves

  • JCo maintains pooled TCP connections to specific SAP app server instances.
  • It has no awareness of SAP-side ZDM, rolling restarts, or HA failovers.
  • It validates connections only conservatively (to avoid latency on every RFC call).
  • It only evicts a connection when an RFC call actually fails on it.

Parameter Reference

Detailed reference for all parameters introduced or modified by this resolution:

Parameter Recommended Value Status Description & Rationale
Connection Topology
jco.client.mshost <message-server> KEEP Hostname of the SAP message server. Enables logon-group-based load balancing across app servers — essential for ZDM resilience.
jco.client.msserv 3601 KEEP Message server port. Standard SAP message server port for client communication.
jco.client.group MULESOFT KEEP Logon group name. JCo asks the message server for a healthy app server within this group at connect time.
jco.client.r3name <SID> ADD SAP system ID (3 chars). Required alongside mshost in most JCo versions to identify the target system. Without it, message-server resolution may fail.
host (ashost) — REMOVE Fixed app-server hostname. Conflicts with mshost — when both are set, JCo typically binds to the fixed host and bypasses load balancing entirely, defeating ZDM resilience.
Connection Pool Behavior — Key Fixes
jco.destination.pool_check_connection ★ 1 ADD Validates each connection at borrow time with a lightweight ping. If the connection is stale, JCo discards it and provides a fresh one. Most direct mitigation for the ZDM stale-connection symptom. Adds a few ms of overhead per RFC call — generally negligible.
jco.destination.expiration_time ★ 300000 (5 min) ADD Maximum idle time (ms) before a pooled connection is closed by the background reaper. Acts as a secondary safety net so connections cycle naturally even if borrow-time validation is bypassed. Common range: 60000–300000.
jco.destination.expiration_check_period 60000 (1 min) ADD How often (ms) the reaper thread runs to check for expired connections. Should be ≤ expiration_time. Lower values reap faster at minimal CPU cost.
Connection Pool Sizing
jco.destination.pool_capacity 5 ADD Steady-state size of the connection pool. Connections beyond this number are closed when idle. Set based on expected concurrency baseline.
jco.destination.peak_limit 10 ADD Maximum concurrent connections allowed (including beyond pool_capacity). Prevents unbounded connection growth under load spikes. 0 = unlimited (not recommended).
jco.destination.max_get_client_time 30000 (30s) ADD Maximum time (ms) a thread will wait for a connection when the pool is exhausted before throwing an exception. Prevents indefinite blocking under load.
Reconnection & Timeouts
response timeout 60000 (60s) CHANGE Was 30000. RFC call timeout. Increased because the first call after a ZDM event may include re-authentication and session setup, which can briefly exceed 30s.
reconnection.attempts 30 CHANGE Was 3. With a 10s frequency, this extends the initial-connect retry window from ~30 seconds to ~5 minutes — comfortably covering most ZDM bounce windows.
reconnection.frequency 10000 (10s) KEEP Wait between reconnection attempts. Current value is reasonable.
★ marks the two parameters that most directly resolve the stale-connection symptom. All other changes are supporting hardening — predictable pool sizing, longer recovery windows, and corrected topology.

Resolution — Part 2: Application Pattern

Add Retry Around SAP Operations

Configuration alone isn't enough. Even with pool_check_connection, an app server can recycle between the validation ping and the actual RFC call. A retry ensures JCo evicts the bad connection and the next attempt succeeds with a fresh one.

<until-successful maxRetries="3" millisBetweenRetries="2000">
   <sap:invoke-function config-ref="SAP_Config" .../>
</until-successful>

Recovery becomes transparent — no manual restart, no consumer-visible failure.

Three Layers of Defense

1. Borrow-time check

pool_check_connection rejects stale connections before they're used.

2. Idle reaping

expiration_time closes idle connections in the background as a safety net.

3. Failure retry

until-successful recovers transparently if a connection fails mid-call.

Summary

The Issue

JCo's connection pool retains stale connections after SAP zero-downtime maintenance, requiring a Mule application restart to recover.

The Cause

This is inherent JCo behavior — it does not detect SAP-side instance recycling. Without borrow-time validation, expiration tuning, and application-level retry, stale sockets persist indefinitely.

The Fix — Three Layers of Defense

Validate

Enable pool_check_connection to verify connections at borrow time.

Reap

Configure expiration_time so idle stale connections close automatically.

Retry

Wrap RFC calls in until-successful for transparent recovery on failure.

Net result: SAP zero-downtime maintenance becomes truly zero-downtime — for our integrations too.
